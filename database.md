# Database Schema: `comsocwebapp`

The library bypasses ORMs entirely in favor of raw SQL to maximize performance and developer transparency. All queries use highly basic, strictly standard ANSI SQL (e.g., standard `SELECT`, `INSERT`, `UPDATE`, `DELETE`). The schema utilizes universally supported data types (`VARCHAR`, `INTEGER`, `TIMESTAMP`) and avoids dialect-specific extensions to guarantee seamless compatibility across SQLite, MySQL, MariaDB, Oracle, and PostgreSQL. Booleans are represented as `INTEGER` (0 or 1).

## Tables

### 1. `users` Table
Stores both administrators and participants (real and dummy).

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `INTEGER` | Primary Key. |
| `email` | `VARCHAR(255)` | User's email address (nullable for dummy users). |
| `password_hash`| `VARCHAR(255)` | Hashed password. |
| `is_admin` | `INTEGER` | 1 if admin, 0 if participant. |
| `is_dummy` | `INTEGER` | 1 if dummy user (generated by admin), 0 if real. |
| `created_at` | `TIMESTAMP` | Timestamp of user creation. |

### 2. `settings` Table
Defines the parameters of a specific social choice problem instance.

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `INTEGER` | Primary Key. |
| `title` | `VARCHAR(255)` | Name of the election/allocation event. |
| `pref_format` | `VARCHAR(50)` | e.g., 'ranking', 'approval', 'budget', 'points'. |
| `status` | `VARCHAR(50)` | e.g., 'draft', 'open', 'closed'. |
| `budget_limit` | `INTEGER` | Global budget or point limit (if applicable). |

### 3. `options` Table
Represents the items, candidates, or projects available in a setting.

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `INTEGER` | Primary Key. |
| `setting_id` | `INTEGER` | Foreign Key linking to `settings.id`. |
| `name` | `VARCHAR(255)` | Name of the candidate/item/project. |
| `description` | `VARCHAR(1000)`| Detailed explanation of the option. |
| `cost` | `INTEGER` | Cost of the project (default 0 for standard voting). |

### 4. `invitations` Table
Manages access control.

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `INTEGER` | Primary Key. |
| `setting_id` | `INTEGER` | Foreign Key linking to `settings.id`. |
| `token` | `VARCHAR(255)` | Unique alphanumeric invitation code. |
| `is_generic` | `INTEGER` | 1 if link can be reused by multiple users, 0 if single-use. |
| `is_used` | `INTEGER` | 1 if token has been consumed (for non-generic links). |

### 5. `preferences` Table
Stores the individual votes, ranks, or budget allocations. Using an integer for `value` allows this table to universally store ranks (1, 2, 3), approvals (1 or 0), or currency/points (e.g., 500).

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `INTEGER` | Primary Key. |
| `user_id` | `INTEGER` | Foreign Key linking to `users.id`. |
| `setting_id` | `INTEGER` | Foreign Key linking to `settings.id`. |
| `option_id` | `INTEGER` | Foreign Key linking to `options.id`. |
| `value` | `INTEGER` | The user's assigned preference (rank, points, or approval flag). |

### 6. `execution_logs` Table
Stores the results of running a rule. 

| Column Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `INTEGER` | Primary Key. |
| `setting_id` | `INTEGER` | Foreign Key linking to `settings.id`. |
| `rule_name` | `VARCHAR(100)` | Name of the rule used (e.g., 'phragmen', 'envy_free'). |
| `outcome` | `VARCHAR(4000)`| Comma-separated or serialized string of winning `option_id`s. |
| `run_log` | `VARCHAR(4000)`| Plain text log explaining the step-by-step resolution. |

## Query Example

To retrieve all preferences for an active setting (e.g., setting ID `5`) for integration with a solver library:

```sql
SELECT 
    u.id AS user_id, 
    u.is_dummy, 
    o.id AS option_id, 
    o.name AS option_name, 
    p.value
FROM 
    users u
JOIN 
    preferences p ON u.id = p.user_id
JOIN 
    options o ON p.option_id = o.id
WHERE 
    p.setting_id = 5
ORDER BY 
    u.id, p.value DESC;
